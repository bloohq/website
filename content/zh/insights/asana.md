## Asana 简介

Asana 与 [Trello](/insights/trello) 和 [Basecamp](/insights/basecamp) 一样，是项目管理领域的早期先驱之一。它由 Facebook 联合创始人 [Dustin Moskowitz](https://www.linkedin.com/in/dmoskov/) 和前 Facebook CTO（首席技术官兼工程副总裁）[Justin Rosenstein](https://www.linkedin.com/in/justinrosenstein/) 于 2008 年创立。

有趣的是，Dustin 是 Mark Zuckerberg 在哈佛的室友，并在早期帮助编写了这个社交网络的代码！

Justin Rosenstein 曾在 Facebook 担任首席程序员，在那里他推出了"点赞"按钮。在加入 Facebook 之前，他还曾在 Google 担任产品经理，为后来成为 Google Drive 的项目以及最初的 GChat 原型做出了贡献。

Asana 的诞生源于他们对大型组织中经常阻碍生产力的混乱任务流和沟通的挫败感。Asana 从 2008 年成立到 2012 年商业发布大约花了四年时间。这两位创始人旨在创建一个平台，简化工作流程并减少在管理任务上花费的时间，让团队能够专注于核心工作。

快进 16 年，Asana 现在是一家上市公司，在纽约证券交易所交易。

### 多归属

也就是说，Asana 确实有一个强大的王牌。[根据我们的基准测试](/compare)，许多项目管理系统都没有让单个任务同时出现在多个项目中的能力。这种功能，通常被称为"多归属"或"跨项目任务"，使团队能够在不同项目或部门之间更有效地管理工作，而无需复制信息。通过允许任务存在于多个项目中，Asana 确保团队成员可以从任何相关的项目上下文中查看和更新任务，为该工作保持单一的真相来源。

这个功能对于跨越多个团队或计划的任务特别有用，例如跨职能项目或有助于各种目标的任务。例如，营销任务可能与产品发布项目和一般营销活动项目都相关。通过多归属，团队成员可以从任一项目访问和更新任务，所做的任何更改都将反映在任务的所有实例中。这不仅通过消除手动更新任务的多个副本的需要来节省时间，还改善了团队之间的协作和可见性。

## 文件存储

[在管理项目文件方面](/platform/features/file-management)，Asana 有几个可能会阻碍您工作流程的限制。

上传到 Asana 的每个文件都限制为最大 100MB，这对于处理大型文档、高分辨率图像或多媒体文件的团队来说可能会有问题。

此外，Asana 缺乏平台内的集成文件共享功能，迫使用户依赖第三方服务来外部共享文件。这种依赖可能会破坏工作流程并增加复杂性。

Asana 的另一个重大缺点是无法在平台内将文件组织到文件夹中。

没有适当的文件夹结构，管理和检索文件变得繁琐，特别是对于具有大量文档的大型项目。用户通常必须依靠搜索功能，这比组织良好的文件夹系统效率低。虽然 Asana 确实与流行的云存储解决方案（如 Google Drive 和 Dropbox）集成，但在多个平台之间导航可能会分散用户体验。

相比之下，Blue 提供了更强大和用户友好的文件存储方法。

Blue 允许每个文件高达 5GB，轻松容纳更大的文档和媒体文件。

它还具有平台内的集成文件共享功能，通过允许用户公开共享文件而无需切换到第三方服务来简化协作。Blue 支持分层文件夹系统，使用户能够逻辑地组织文件，这简化了导航和文件检索，即使在复杂的项目中也能保持秩序和效率。

通过在平台内提供全面的文件管理功能，Blue 确保了更流畅、更一致的用户体验，提高了生产力并减少了摩擦。对于管理大量或大型文件的团队，Blue 的方法提供了明显的优势。处理更大文件、有效组织它们并无缝共享它们的能力可以显着提高工作流程效率。

相比之下，Asana 的限制可能需要额外的工具和变通方法，可能导致工作流程分散和生产力降低。对于优先考虑精简和集成文件管理系统的团队，Blue 作为 Asana 的一个引人注目的替代方案脱颖而出。

### 仪表板

Asana 和 Blue 都有仪表板，但当您深入了解细节时，您会意识到 Asana 实施仪表板的方式存在严重缺陷。

创建准确且有影响力的仪表板所需的关键事项之一是能够精确筛选到您需要的数据。否则，您将被迫导出数据并使用第三方系统，如 Metabase 或 PowerBI。

Asana 仪表板在使用自定义字段进行筛选时有几个限制。

自定义字段筛选是可用的，但并非所有字段类型都得到同等支持。

单选自定义字段可用于筛选，**而多选、基于文本和一些数字自定义字段在仪表板中的筛选功能有限或没有**。

此外，只有组织范围的自定义字段可用于组合级别或全局报告中的筛选，这对于使用项目特定自定义字段的项目可能会有问题。

此外，在仪表板中使用带有自定义字段筛选器的"任何"或"非"条件存在限制，这与搜索筛选器不同。这可能使创建更复杂或包容性的筛选器变得困难。

用户还报告了某些自定义字段作为筛选选项的可用性不一致，一些字段在图表中显示为轴但不作为筛选器。这些限制可能会显着影响通过 Asana 仪表板进行分析的灵活性和深度。

Blue 采用了完全不同的方法。[我们在项目视图上有高级筛选器](https://documentation.blue.cc/views/filters#advanced-filters)。创建仪表板时，您可以利用*完全相同的筛选器*。

## API 和 Webhooks

### API

Asana 提供完整的 API 访问，速率限制为每分钟 1,500 个请求。此 API 允许您访问所有端点，包括关键的"搜索"端点。但是，在容器（如任务、故事或项目）中读取最多 50,000 个项目存在限制。API 使用 RESTful 接口，支持请求中的 JSON 或表单编码内容，并返回 JSON 响应。

Asana 为各种编程语言（如 Node.js、Java 和 PHP）提供客户端库，使开发人员能够读取信息、输入数据并自动对更改做出反应。对于身份验证，Asana 使用 OAuth 2.0（Asana Connect），还支持用于个人访问和开发的个人访问令牌。API 包括安全措施，如 CORS 标头、请求签名验证和过期时间检查。

但是，Asana 的 API 有一些批评。

某些功能（如规则）目前无法通过 API 访问。此外，一些 API 限制没有很好的文档记录，给开发人员带来困难。访问子任务可能效率低下，并可能导致速率限制问题，因为 API 没有提供直接的方法来确定任务是否有子任务而不进行额外的请求。

此外，API 缺乏有效的方法来获取诸如"subtask_count"或"attachments_count"之类的信息，而无需额外的请求。

对于免费计划的用户，API 限制为每分钟 150 个请求，并且不提供对"搜索任务"端点的访问。这可能会限制不订阅付费计划的用户可用的功能。

[Blue GraphQL API](/platform/api) 100% 覆盖了 Blue 中的所有功能。

因此，您可以通过我们的界面执行的*任何*操作都可以以编程方式触发。这是我们首次创建 API 时的有意设计选择，我们希望让开发人员完全访问所有内容。

每次我们[发布新功能](/platform/changelog)时，API 都会更新以支持这些新功能。

### Webhooks

Webhooks 允许应用程序实时收到 Asana 中更改的通知。它们的工作方式类似于 Asana 的事件资源，但通过 HTTP POST 将事件"推送"到指定的 URL，而不是需要轮询。

Webhooks 可以在不同级别创建：项目、组合、团队或工作区。但是，有一些重要的范围限制需要考虑。任务级别的事件，包括子任务和故事，不会传播到更高级别的 webhooks，如工作区级别的 webhooks。工作区级别的 webhooks 需要至少一个筛选器，所有筛选器必须在 Asana 的白名单中。

Asana webhooks 执行定期健康检查，称为"心跳事件"或"健康检查"。当 webhook 由于健康检查失败而变为非活动状态时，没有内置的通知系统来提醒用户。

Asana webhooks 提供细粒度的配置选项，允许您根据特定需求定制通知。您可以为单个项目、组合、团队或工作区创建 webhooks，建议为与任务相关的事件使用项目级别的 webhooks。

系统通过 WebhookFilter 对象支持详细的事件筛选。这使您能够指定哪些资源类型（例如，任务、故事）和操作（添加、更改、删除）触发通知。对于任务，您甚至可以筛选特定字段，例如截止日期更改。

多个筛选器可以组合在单个 webhook 配置中，充当白名单，其中任何匹配的事件都会触发 webhook。这种灵活性允许对 webhook 通知进行精确控制，从广泛的项目范围更新到特定的任务字段更改。

Blue 提供编程访问*和* Web 界面来创建、查看和管理 Webhooks。

Blue 的 webhook 选项还提供了更细粒度的方法，允许用户选择特定事件来触发通知。这种定制级别包括对记录、清单、自定义字段、标签和评论等的操作。这种粒度确保您可以根据您的确切需求定制 webhooks，只在最相关的事件上触发。

此外，Blue 的 webhooks 可以按一个或多个项目进行筛选，甚至跨不同的组织或工作区。这意味着您可以创建复杂的跨项目自动化规则和通知，增强各个团队和部门之间的协调和生产力。通过用户友好的界面管理这些设置的能力简化了流程，即使对于没有编程专业知识的人也可以访问。

## 用户界面和体验

Blue 的突出功能之一是能够免费使用多个自定义域，包含在订阅中。这意味着用户可以在自己的域或子域上访问 Blue。这对代理机构或专业服务特别有利，因为它允许他们展示自己的品牌而不是 Blue 的品牌。相比之下，Asana 缺乏此功能，这意味着用户始终暴露于 Asana 的品牌。对于希望保持一致和专业品牌形象的企业来说，这可能是一个缺点。

### 布局和可用性

Asana 的界面在过去 16 年中已经更新，但它仍然保留了一些遗留的设计元素，使其与 Blue 相比感觉不太现代。一个值得注意的问题是任务只从右侧滑入，这可能会感到狭窄和低效，特别是在同时管理多个任务时。这种设计选择通常会导致用户体验不够流畅。

此外，Asana 缺乏在侧边栏中拖放重新排序项目的能力，这是一个增强可用性和组织的基本 UX 功能。这种遗漏对于需要快速重新排列项目的用户来说可能会令人沮丧。

另一个重要的限制是 Asana *只允许一个人被分配到一个任务。* 虽然这可能符合 Asana 对任务管理的哲学方法，但它限制了灵活性。另一方面，Blue 允许将多个用户分配到一个任务。

总体而言，Blue 的用户界面在设计时考虑了现代性和灵活性，确保了更直观和用户友好的体验。用户欣赏精简的设计以及自定义工作区以更好地适应其品牌和组织需求的能力。相比之下，Asana 的界面虽然功能齐全，但由于其设计选择和限制，可能会感到过时和不够灵活。

总之，虽然两个平台都有其优势，但 Blue 专注于可定制、现代的界面和增强的可用性功能，使其成为寻求高效且品牌一致的项目管理工具的团队的引人注目的选择。

## 公司结构的差异

当您选择项目管理工具时，重要的是不仅要看其功能。产品背后的公司也很重要。他们的公司结构和重点可能会影响工具的发展及其长期可持续性。

让我们从 Asana 开始。

Asana 是一家上市公司，在纽约证券交易所上市。它有 1,840 名员工分布在 13 个办公室：

1. 旧金山，加州（总部）
2. 芝加哥，伊利诺伊州
3. 纽约市，纽约州
4. 悉尼，澳大利亚
5. 温哥华，加拿大
6. 巴黎，法国
7. 慕尼黑，德国
8. 雷克雅未克，冰岛
9. 都柏林，爱尔兰
10. 东京，日本
11. 华沙，波兰
12. 伦敦，英国
13. 新加坡

因为 Asana 是一家上市公司，我们可以深入研究其财务状况，看看它是否可持续。这很重要，因为一家不可持续的公司可能会让您的项目面临风险，或者至少他们很可能在未来提高价格以试图弥补损失。

在上市之前，Asana 在 5 轮投资中筹集了 4.53 亿美元。

自从在纽约证券交易所上市以来，从 2021 年 11 月至今，Asana 的价值已经下跌了 89.29%。它的交易价格为 142.68 美元，现在交易价格约为 15 美元。

Asana 自其历史最高点以来股价大幅下跌的主要原因是[项目管理软件市场的激烈竞争](/solutions/use-case/project-management)，这些竞争对手提供类似或可能更好的价值主张。

损失已经累积！从 2020 年到 2024 年的总损失达到 4.433 亿美元。

| 年份 | 损失（百万美元） |
|:----:|:------------:|
| 2020 |    118.6     |
| 2021 |     98.9     |
| 2022 |     58.7     |
| 2023 |     99.2     |
| 2024 |    67.9*     |

*注：2024 年的数字是 GAAP 运营损失，而不是全部净损失，可能不代表整个财年。

这是因为 Asana 不得不在销售和营销上花费巨额资金（[这是他们的年度报告](https://investors.asana.com/node/9876/pdf)）。

**仅在 2023 年，他们就在销售和营销上花费了 4.349 亿美元。**

😳

这项投资带来了 1.69 亿美元的收入增长。

为了衡量这笔支出的有效性，我们可以计算收入效率比：

收入增长 / 销售和营销支出

计算数字，Asana 在销售和营销上每花费 1 美元，只产生了 0.39 美元的新收入。

这些比率表明，Asana 2023 年的销售和营销支出在产生新收入方面并不高效。通常，高于 1 的比率被认为是良好的，表明公司产生的新收入超过了在销售和营销上的支出。

显然，对于 B2B SaaS（软件即服务），事情比这要复杂一些，但这是一个很好的经验法则。

这对潜在客户来说很重要，因为您想知道您的供应商的重点在哪里。

与他们在 2023 年 4.343 亿美元的营销支出相比，他们在产品开发上花费了 2.97 亿美元——所以我们让您自己做决定。

如果您是一家小企业，您还应该考虑一家拥有 13 个办公室和 1,800 多名员工的公司对您面临的问题真正了解多少？

实际上，他们对在风险投资硅谷泡沫之外的世界中经营可持续业务*一无所知*。
